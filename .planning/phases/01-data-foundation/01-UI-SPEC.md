## Scope Note

Phase 1 is a **Python data pipeline phase**. It produces no browser UI. The only rendered outputs are:

1. **CLI progress logging** — printed to stdout during the pipeline run
2. **Data quality report** (DATA-05) — a tabular summary of review counts by neighbourhood and by year, used as a gate before Phase 2 begins

All frontend UI (map, sidebar, legend, time slider) is built in Phases 4–5. The design contract below applies exclusively to these two pipeline-output surfaces.

## CLI Progress Logging Contract

The pipeline emits progress to stdout. All log lines follow a fixed format so grep and automated parsing work cleanly.

### Log Line Format

```
[LEVEL] YYYY-MM-DD HH:MM:SS  message
```

Levels (fixed-width, uppercase, 4 chars):

| Level | When to Use |
|-------|-------------|
| `INFO` | Normal progress milestones (file opened, batch written, join complete) |
| `WARN` | Non-fatal issues (record skipped, missing lat/lng, boundary edge case) |
| `FAIL` | Fatal error — pipeline cannot continue |

### Progress Milestones to Emit (INFO)

Emit at minimum these milestones — executor must include all of them:

1. `Yelp business file open — streaming parse started`
2. `NYC businesses identified: {n} records` (after filtering to NY state)
3. `Spatial join complete — {n} businesses assigned to neighbourhoods`
4. `Review file open — streaming parse started`
5. `Reviews written: {n}` — emit every 100,000 records
6. `Import complete — {n} total reviews in database`
7. `Data quality report written: {path}`

### WARN Line for Skipped Records

```
[WARN] YYYY-MM-DD HH:MM:SS  Skipped business {business_id}: missing lat/lng
```

Do not emit a WARN per record if volume exceeds 1,000 warnings — switch to a summary count line instead:

```
[WARN] YYYY-MM-DD HH:MM:SS  {n} records skipped for missing lat/lng (suppressed individual warnings)
```

## Data Quality Report Contract

The data quality report satisfies DATA-05. It is a **Markdown file** written to `data/quality_report.md`. It must be human-readable in a terminal (`cat`) and renderable on GitHub.

### Required Sections (in order)

#### 1. Summary Header

```
# Data Quality Report
Generated: YYYY-MM-DD HH:MM UTC
Source: Yelp Open Dataset
```

#### 2. Overall Counts Table

| Field | Value |
|-------|-------|
| Total reviews imported | {n} |
| Unique businesses | {n} |
| Neighbourhoods covered | {n} of {total NTAs} |
| Date range | {earliest review date} to {latest review date} |

#### 3. Coverage by Neighbourhood (table)

Columns: `Neighbourhood | Borough | Total Reviews | 2019 | 2020 | 2021 | 2022 | 2023 | 2024 | Years with Coverage`

Sort: descending by Total Reviews.

**Gate threshold** (inline in report — highlight with bold if failing):

> **GATE**: Every neighbourhood must have reviews in at least 5 of 7 years (2019–2025) and at least 500 total reviews. Rows failing this gate are marked **FAIL**.

Mark failing rows with `**FAIL**` in the `Years with Coverage` column. If any rows fail, emit a summary line at the end of this section:

```
**Coverage gate: X neighbourhoods FAIL. Do not proceed to Phase 2.**
```

If all pass:

```
Coverage gate: PASS. All neighbourhoods meet minimum thresholds.
```

#### 4. Skipped Records Summary

| Reason | Count |
|--------|-------|
| Missing lat/lng | {n} |
| Outside Manhattan/Brooklyn NTA polygons | {n} |
| Duplicate business_id (INSERT OR IGNORE) | {n} |
| Review timestamp unparseable | {n} |

#### 5. Phase 2 Readiness

Single line verdict:

```
READY FOR PHASE 2  ✓
```
or
```
NOT READY — resolve coverage failures above before proceeding.
```

## Color

Not applicable to CLI output or Markdown report.

The only colour convention is terminal ANSI output for the pipeline logger:

| Level | ANSI Color | Hex Equivalent |
|-------|-----------|----------------|
| INFO | default (no color) | — |
| WARN | yellow | `#FFC107` |
| FAIL | red | `#F44336` |

Apply ANSI codes only when stdout is a TTY (`sys.stdout.isatty()` check). Never emit ANSI codes to piped output or log files.

Accent reserved for: WARN and FAIL log levels only.

## Copywriting Contract

Phase 1 has no user-facing UI copy. The copywriting contract covers the quality report text and CLI messages only.

| Element | Copy |
|---------|------|
| Primary CTA | Not applicable — no interactive UI in this phase |
| Empty state (no reviews found) | `[FAIL] {timestamp}  No NYC reviews found in dataset. Validate that yelp_academic_dataset_review.json is the correct file and re-run.` |
| Error state (file not found) | `[FAIL] {timestamp}  Dataset file not found at {path}. Set YELP_DATA_DIR environment variable to the directory containing the Yelp Open Dataset files.` |
| Error state (SQLite write failure) | `[FAIL] {timestamp}  Database write failed: {sqlite_error}. Check disk space and file permissions at {db_path}.` |
| Coverage gate failure | `NOT READY — resolve coverage failures above before proceeding.` |
| Coverage gate pass | `Coverage gate: PASS. All neighbourhoods meet minimum thresholds.` |
| Phase 2 readiness pass | `READY FOR PHASE 2  ✓` |
| Phase 2 readiness fail | `NOT READY — resolve coverage failures above before proceeding.` |
| Destructive confirmation | Not applicable — no destructive user actions in this phase |

## Interaction Contract

Phase 1 has no interactive UI. The only "interactions" are:

| Trigger | Response |
|---------|----------|
| User runs `python ingest.py` | Pipeline starts, emits INFO log lines, writes DB, writes quality report |
| User runs `python ingest.py` a second time | Idempotent — `INSERT OR IGNORE` prevents duplicates, re-writes quality report with current counts |
| Missing env var / file | Emits `[FAIL]` log line with exact path and resolution instruction; exits with code 1 |
| KeyboardInterrupt (Ctrl-C) | Pipeline stops cleanly; partial DB state is valid (idempotent on resume) |
